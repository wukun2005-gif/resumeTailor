# 软件开发工作原则 (Long-term Specification)

0. 未经我同意，不能改动任何代码源文件和文档。
1. 任何修改必须基于对现状的充分理解，且在不破坏既有功能和接口契约的前提下进行；涉及破坏性变更时须在 PR/提交说明中显式标注 BREAKING。
2. 必须先看项目现在的代码和文档还有历史change log，全面深刻地理解项目的需求、设计、测试和既往改动后，才能开始工作。
3. 新功能必须补充对应测试，沿用项目现有测试框架（如 `test-e2e.mjs`、`test-filter.js` 所用体系），按职责归类放入对应测试文件或新建同体系文件；至少覆盖正常路径与关键边界，核心逻辑需有单元测试，跨模块流程需有集成/E2E 测试。
4. 任何改动都须先设计测试方案再实施；提交前必须本地执行相关新测试，以及全部相应的既有测试，全部通过方可提交；失败用例不得通过 skip/注释绕过。

4.1. **测试分析与执行策略**：
   - 改动完成后，必须分析影响范围，确定需要运行的测试集：
     - **API/后端改动**：必须运行完整的 E2E 测试（`test-e2e.mjs`），确保 API 契约未被破坏。
     - **前端逻辑改动**：若涉及状态管理、事件处理、API 调用等，需运行 E2E 测试验证 API 层面行为；若涉及纯 UI 交互（如持久化、事件绑定），需说明现有测试的覆盖范围与遗漏点，并建议手动验证方案或新增浏览器自动化测试。
     - **模板/Prompt 改动**：必须运行模板相关的单元测试和 E2E 测试中的对应用例。
   - 测试执行结果必须明确告知用户：通过了哪些测试，覆盖了什么范围。
   - 任何测试的"跳过"或"豁免"必须说明理由，不得以"构建成功"代替功能测试。
5. 任何涉及行为、接口、架构或使用方式的代码改动，必须同步更新 `DESIGN.md` 与 `README.md` 的相关小节；变更记录以 `YYYY-MM-DD | 简述 | 影响范围 | 关联 commit` 的条目形式追加到 `DESIGN.md` 的 `Change Log` 章节。纯注释、格式化、重命名等无行为变化的改动可豁免，但仍需在 commit message 中说明。
6. 当任务需要读取 PDF 时，先查阅 `/Users/wukun/Documents/jl/AGENTS.md` 中的本地读取方案并据此实现，不要自行引入新的 PDF 解析依赖。
7. 工作中生成的各种临时文件、临时安装的工具程序、临时的测试代码等等，改动后要清理干净，不要留下垃圾：调试/实验用的临时文件和脚本一律删除；临时安装的全局工具须卸载，临时的项目依赖从 `package.json` 中移除；不应提交的产物加入 `.gitignore`；交付前用 `git status` 核对工作区干净，仅保留与本次任务直接相关的改动。
8. 优先级与冲突仲裁。当以下原则之间发生冲突时，按由高到低的优先级处理，并在提交说明中注明所做的权衡：若时效与前四项冲突，不得通过跳过测试、删减测试、降级校验、注释掉失败断言等方式抢时间；正确做法是：缩小本次改动范围、拆分为多个 PR、或显式向用户请示延期。任何偏离该优先级的决定都必须先经用户确认。
   8.1. 安全与数据完整性（不得造成数据丢失、密钥泄露、生产事故）
   8.2. 正确性（既有与新增测试全部通过）
   8.3. 文档与代码一致性（`DESIGN.md` / `README.md` 与实现同步）
   8.4. 代码质量与可维护性（可读性、最小改动、遵循既有架构）
   8.5. 交付时效
9. 禁止项（Hard Rules，违反即回滚）
   9.1. 禁止伪造或篡改测试结果（包括手动修改测试输出、伪造覆盖率数据、mock 掉本应真实验证的行为）。
   9.2. 禁止通过 `skip`、`xit`、`it.only` 永久屏蔽失败用例，或注释/删除失败用例来"让测试通过"；确需临时跳过必须在用例上方写明原因、关联 issue 编号，并同步告知用户。
   9.3. 禁止在未经用户显式确认的情况下执行破坏性 git 操作，包括但不限于：`git push -f` / `--force-with-lease`、`git reset --hard`、`git clean -fdx`、`git rebase` 已推送分支、删除分支/tag、重写历史（`filter-branch`、`filter-repo`）、`git checkout -- .` 覆盖未提交改动。
   9.4. 禁止直接向 `main` / `master` / `release/*` 等受保护分支强推或直接 commit；改动走分支 + PR 流程。
   9.5. 禁止私自引入新的运行时依赖、付费服务、外网调用、遥测上报；确需引入，先在沟通中说明用途、许可证、体积与替代方案，待用户批准。
   9.6. 禁止提交密钥、令牌、`.env`、个人隐私数据；发现历史中已混入须立即告警并走吊销流程，不自行 `filter-branch`。
   9.7. 禁止在未备份的前提下删除或覆盖用户文件、数据库、迁移脚本。
10. 沟通纪律（先确认，后动手）。遇到以下情形必须先用一条简短消息向用户说明方案与风险，获得确认后再实施，不得"先做了再说"：
    10.1. 方案选型：在两种及以上实现路径、库/框架、架构模式之间抉择时。
    10.2. 接口变更：对外 API、CLI 参数、配置项、数据库 schema、文件格式的新增/修改/删除。
    10.3. 跨模块重构：影响 3 个及以上文件或跨越既有模块边界的重构。
    10.4. 新增依赖或基础设施：新包、新服务、新环境变量、新端口、新后台任务。
    10.5. 删除或大规模移动既有代码/文件/文档。
    10.6. 预计改动量超过约 200 行，或预计耗时超过单次任务合理范围。
    10.7. 任何本原则清单中标注"需用户确认"的操作。确认消息应包含：①目标与动机；②候选方案对比（至少两个，含"不做"选项）；③推荐方案及理由；④影响范围与回滚方式。用户明确回复后方可进入实施；仅口头"继续"视为同意推荐方案，歧义时再次澄清。
